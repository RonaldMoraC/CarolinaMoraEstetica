1. Skill de Tipado Estricto y Arquitectura Limpia (Clean Code):
Directiva: "Actúa como un compilador humano de PHP 8.2+. Todo archivo generado debe iniciar obligatoriamente con declare(strict_types=1);. Es mandatorio aplicar tipado estricto a todas las propiedades de las clases, parámetros de métodos y tipos de retorno (return types), incluyendo el uso de tipos nulos (?string, ?int) o tipos unión (string|int) cuando sea necesario."

Patrón Obligatorio y Desacoplamiento: "Separa estrictamente las capas del sistema. Las clases de Domain no pueden instanciar ni hacer referencia a nada de Infrastructure. Toda comunicación entre capas debe realizarse exclusivamente mediante Interfaces (Contratos) inyectadas a través del constructor de la clase (Constructor Injection), prohibiendo el uso de variables globales, singletons rígidos o funciones mágicas de frameworks."

Por qué importa: Elimina por completo los errores invisibles de datos en el servidor. Al obligar a la IA a tipar los retornos y usar inyección por constructor, el código se vuelve 100% testeable, modular y listo para la futura migración a Laravel o Microservicios sin tener que reescribir la lógica de negocio.

2. Skill de Programación Defensiva contra Concurrencia (DBA Senior):
Directiva: "Al escribir la lógica de verificación y creación de reservas (Appointment), asume un entorno de alta concurrencia omnicanal. Toda validación de disponibilidad debe ejecutarse dentro de una transacción PDO con un nivel de aislamiento mínimo de REPEATABLE READ. Prohibido usar SELECT simples; debes implementar bloqueos pesimistas utilizando de forma explícita SELECT ... FOR UPDATE apuntando estrictamente a los índices de la mallas horarias (work_schedule) y rangos comprometidos. Debes agregar la cláusula NOWAIT o un manejo estricto de excepciones para capturar el error de 'Lock Wait Timeout' (Código SQLState 40001 / Deadlock), asegurando que cualquier fallo ejecute un ROLLBACK inmediato y libere las conexiones."

Por qué importa: Cuando el Bot de WhatsApp y la PWA compitan por la misma fracción de segundo, el primer proceso bloqueará la ranura de tiempo. El segundo proceso, gracias a esta skill, no se quedará colgado "congelando" el servidor de Hostinger; detectará el bloqueo al instante, abortará la operación de forma segura (ROLLBACK) y le notificará al usuario que "el horario ya ha sido tomado por otro cliente", eliminando el overbooking y los colapsos de base de datos.

3. Skill de Optimización para Entornos Compartidos (Hostinger Premium):
* Directiva: "Estás programando para un entorno Hostinger Premium (recursos de CPU y RAM compartidos). Queda estrictamente prohibido el uso de funciones que requieran privilegios de root, demonios persistentes (systemd) o bucles infinitos 'while(true)'. Los workers de colas ('NotificationQueueWorker.php') deben ser diseñados como scripts por lotes (Batch CLI) autolimitados. El script debe: 1) Implementar un mecanismo de semáforo mediante un archivo de bloqueo físico ('flock') al iniciar, para impedir que dos ejecuciones del Cron se solapen. 2) Auto-limitar su ejecución a un máximo de 45 segundos o un lote controlado de N registros. 3) Invocar explícitamente la liberación de memoria ('unset') y cerrar las conexiones PDO activas al finalizar el lote, garantizando una muerte limpia del proceso."
* Por qué importa: Evita de forma absoluta la suspensión de la cuenta de hosting de la Estética Carolina Mora por exceso de uso de CPU. Al usar un bloqueo por archivo ('flock'), aseguras que un mensaje de WhatsApp jamás se envíe duplicado si el servidor se ralentiza, manteniendo el sistema ligero, rápido y dentro de las políticas de uso de Hostinger.

4. Skill de Manejo de Errores Estandarizado (API REST):
* Directiva: "Cualquier excepción atrapada en la capa de controladores o el Front Controller tiene estrictamente prohibido devolver un 'echo', un 'var_dump', un bloque de texto plano o código HTML genérico. Toda respuesta de error debe formatearse bajo la especificación estricta RFC 7807 (Problem Details). El script debe: 1) Forzar el encabezado HTTP 'Content-Type: application/problem+json'. 2) Devolver un payload JSON que contenga exactamente las claves: 'type' (URI del error), 'title' (resumen corto), 'status' (código HTTP coincidente con el header), 'detail' (explicación humana) e 'instance' (URI de la petición). 3) Implementar una guardia de entorno basada en el archivo '.env': si el entorno es producción ('APP_ENV=production'), se deben omitir detalles internos del sistema de archivos o trazas de la base de datos, sustituyéndolos por un mensaje seguro y registrando la traza real de forma asíncrona en los logs PSR-3 del servidor."
* Por qué importa: Garantiza que la PWA, la aplicación móvil y el Webhook de WhatsApp consuman una estructura de errores idéntica y predecible. Esto evita caídas imprevistas en las interfaces cliente ante caídas de servidor y blinda la infraestructura de Hostinger contra fugas de información que puedan ser aprovechadas por atacantes para hackear la base de datos.


5. Skill de Parseo y Resiliencia para Webhooks (WhatsApp Bot):
* Directiva: "El controlador encargado de recibir los eventos del Webhook de WhatsApp ('WhatsAppWebhookController.php') debe operar bajo un patrón asíncrono desacoplado y blindado. El script debe realizar estrictamente los siguientes pasos en orden secuencial inmediato: 1) Validar la autenticidad del remitente verificando la firma criptográfica 'X-Hub-Signature-256' contra el 'WHATSAPP_APP_SECRET' del archivo '.env'. 2) Validar el handshake de verificación inicial (modo 'subscribe' de Meta). 3) Leer el JSON entrante y extraer el ID único del mensaje ('wamid'). Si el 'wamid' ya existe en la tabla de auditoría o logs, abortar de inmediato para prevenir duplicados. 4) Insertar el payload crudo en la tabla física de cola de mensajes pendientes ('wa_notification_queue') con estado 'PENDING'. 5) Forzar el cierre inmediato de la conexión HTTP enviando un código 'HTTP 200 OK' a Meta mediante el vaciado explícito del búfer de salida ('fastcgi_finish_request()' o cabeceras de conexión equivalentes), deteniendo el flujo web de Meta antes de iniciar cualquier lógica conversacional o consulta de base de datos."
* Por qué importa: Elimina radicalmente el riesgo de bucles infinitos de retransmisión por parte de los servidores de Meta ante picos de tráfico. Al cerrar la conexión HTTP en milisegundos tras registrar el mensaje crudo, la base de datos de la Estética Carolina Mora queda protegida contra la saturación de conexiones abiertas y se garantiza que el procesamiento pesado de las mallas horarias ocurra en segundo plano por el Batch Worker sin ralentizar la API.


6. Skill de Arquitectura Idempotente (Pasarelas de Pago y POS):
* Directiva: "Toda acción de cobro, mutación de facturas o procesamiento de Webhooks financieros en la capa de 'Billing' debe ser estrictamente idempotente. Para garantizarlo, el sistema debe: 1) Exigir u obtener una Clave de Idempotencia única (compuesta por el ID de la transacción externa o un hash del request 'idempotency_key'). 2) Implementar un bloqueo distribuido o registro atómico previo en una tabla de control indexada. Si la clave no existe, se inserta con estado 'PROCESSING'. Si existe y su estado es 'PROCESSING', abortar inmediatamente devolviendo un código 'HTTP 409 Conflict'. 3) Envolver todo el proceso de afectación de las tablas de facturas ('invoice') y pagos ('payment') dentro de una transacción PDO aislada. 4) Si la clave ya existía con estado 'SUCCESS', el controlador no debe procesar nada ni lanzar un error; debe recuperar el payload de respuesta guardado originalmente en la base de datos y retornar exactamente esa misma respuesta con un código 'HTTP 200 OK'."
* Por qué importa: Blinda financieramente el ecosistema de la Estética Carolina Mora contra cobros duplicados y descuadres de caja en el POS provocados por reintentos de red automáticos o dobles clics accidentales de las clientas. Al cachear las respuestas de éxito y bloquear las peticiones concurrentes en estado de procesamiento, se asegura la consistencia matemática absoluta del módulo contable en Hostinger.


7. Skill de Gestión Inmutable del Tiempo y Fechas (Time-Zone Guard):
* Directiva: "Al manipular calendarios, agendas, bloques horarios o excepciones de asistencia en las capas de 'Staffing' y 'Booking', queda estrictamente prohibido el uso de funciones de tiempo nativas implícitas (como 'date()' o 'time()') o la instanciación de la clase mutable 'DateTime'. El sistema debe: 1) Procesar toda fecha, hora o intervalo exclusivamente mediante la clase 'DateTimeImmutable' y 'DateTimeZone', forzando el huso horario explícito 'America/Bogota'. 2) Almacenar y recuperar la información en la base de datos garantizando que no existan conversiones implícitas a UTC por parte del controlador PDO; toda consulta que filtre rangos horarios debe formatear el string de comparación bajo el estándar 'Y-m-d H:i:s'. 3) Ignorar por completo la zona horaria del dispositivo cliente (PWA) durante los cálculos lógicos de disponibilidad en el backend, procesando las restricciones operativas basándose únicamente en el tiempo de la sucursal física correspondientemente afectada."
* Por qué importa: Evita el desplazamiento o la corrupción de los horarios de las citas de la Estética Carolina Mora debido a desfases entre el servidor y las bases de datos. Al usar objetos inmutables, se asegura que los bucles que renderizan el calendario diario de los especialistas no alteren accidentalmente los datos en memoria, eliminando errores donde las citas se mueven solas de hora.

8. Skill de Sincronización Estado-Cliente (Offline-First para PWA):
* Directiva: "Al programar los scripts de JavaScript de la PWA, debes diseñar una arquitectura asíncrona robusta utilizando 'IndexedDB' para persistencia local y un 'Service Worker' central. Toda mutación de estado (como el agendamiento de una cita) intentada sin conexión de red debe: 1) Generar un token de correlación UUID temporal y almacenar el payload crudo en un almacén de objetos ('Object Store') de IndexedDB estructurado como una cola de despacho ('outbox_queue'). 2) Actualizar inmediatamente la interfaz gráfica mostrando la acción en estado 'Pendiente de Sincronización' para evitar la frustración del usuario. 3) Registrar un evento de sincronización utilizando la API nativa 'Background Sync' del Service Worker ('sw.js'). Al recuperar la conectividad, el Service Worker procesará la cola enviando las peticiones a la API REST. Si el servidor rechaza la transacción por falta de disponibilidad horaria (Conflicto), el Service Worker debe enviar un mensaje al hilo principal a través de 'BroadcastChannel' para notificar al usuario de forma amigable y permitirle elegir un nuevo horario."
* Por qué importa: Garantiza que la aplicación de la Estética Carolina Mora mantenga una operatividad ininterrumpida incluso en sótanos, zonas con mala cobertura o micro-cortes de señal. Al delegar la sincronización en Background Sync de forma asíncrona, se asegura que los datos jamás se pierdan y que la interfaz responda instantáneamente, eliminando pantallas de error invasivas.

* Por qué importa: Provee un historial transparente e inalterable ante cualquier disputa financiera o de asignación de turnos en la Estética Carolina Mora. Al guardar los estados JSON de antes y después de cada cambio, el administrador puede reconstruir con precisión forense qué empleado modificó un registro, en qué milisegundo ocurrió y qué dato exacto fue alterado, blindando el negocio contra fraudes internos.

10. Skill de Sanitización Perimetral Absoluta (SecOps / Inyección):
* Directiva: "Aplica el principio de desconfianza cero (Zero-Trust) en la arquitectura de datos. Para neutralizar ataques de inyección SQL (incluyendo los de segundo orden) y de scripts (XSS), el sistema debe: 1) Ejecutar un pipeline obligatorio de validación, tipado estricto y casteo explícito en los controladores para todo dato entrante desde la API, formularios o Webhooks antes de tocar los Casos de Uso. 2) Queda terminantemente prohibido el uso de funciones de escape manual de strings o la mutación con 'htmlspecialchars' antes de guardar los datos en la base de datos; los textos deben persistirse en su estado original e íntegro. 3) Toda interacción con la base de datos MySQL en los repositorios PDO debe procesarse exclusivamente mediante Sentencias Preparadas (Prepared Statements) con mapeo de parámetros ('bindValue'), garantizando que ningún dato se concatene directamente en las sentencias SQL. 4) La prevención contra ataques XSS debe delegarse de forma obligatoria y exclusiva a la capa de salida (Output), aplicando funciones de sanitización y escape de caracteres en el renderizado del frontend o inmediatamente antes de emitir cualquier respuesta JSON en la API."
* Por qué importa: Blinda el sistema de la Estética Carolina Mora contra accesos no autorizados y robo de datos sin dañar la integridad de la información (como nombres con tildes, comillas o emojis de WhatsApp). Al forzar sentencias preparadas en el 100% de las consultas y sanitizar los datos estrictamente en la salida, se anula la capacidad de un atacante de secuestrar la sesión de la recepcionista mediante código malicioso inyectado en las notas de las citas.

11. Skill de Estrategias de Reintento con Respaldo Exponencial (Exponential Backoff):
* Directiva: "Toda integración con servicios de red externos (como Meta Cloud API o Pasarelas de Pago) debe encapsularse bajo un patrón de resiliencia ante errores transitorios (HTTP 5xx, timeouts o pérdidas de paquetes). Queda estrictamente prohibido el uso de funciones de bloqueo de ejecución web como 'sleep()' o 'usleep()'. En su lugar, el sistema debe operar mediante una persistencia asíncrona en base de datos bajo las siguientes reglas: 1) Cada fallo detectado debe incrementar un contador 'attempts' en la tabla de colas correspondiente y cambiar el estado del registro a 'RETRY_PENDING'. 2) El tiempo del próximo intento debe calcularse mediante la fórmula matemática de respaldo exponencial: $Tiempo = Base \times 2^{attempts} + Jitter$, donde la 'Base' inicial es de 2 segundos y el 'Jitter' introduzca una desviación aleatoria de milisegundos para romper la sincronización de peticiones. 3) Se debe establecer un límite estricto de máximo 5 intentos. Si el contador supera este límite, el registro debe mutar permanentemente al estado 'FAILED' y registrar un evento de criticidad alta en el log del sistema."
* Por qué importa: Asegura que el Bot de WhatsApp y el motor de cobros de la Estética Carolina Mora toleren micro-caídas de internet o pestañeos de servidores externos sin perder datos ni colgar el servidor de Hostinger. Al delegar el cálculo del tiempo a una columna 'next_retry_at' leída por el Batch Worker en segundo plano, la aplicación web sigue respondiendo de inmediato a las usuarias mientras el sistema gestiona la recuperación en frío..


12. Skill de Hidratación de Entidades Eficiente (Performance DBA):
* Directiva: "Al implementar los repositorios PDO para mapear las filas de la base de datos a objetos de dominio (entidades), queda terminantemente prohibido ejecutar consultas SQL secundarias dentro de ciclos o bucles iterativos (Problema del N+1). Para resolver la carga de relaciones en colecciones de datos, el sistema debe: 1) Evaluar el tipo de relación; para dependencias directas uno a uno, utilizar JOINs SQL explícitos indexados. 2) Para colecciones masivas o relaciones complejas, aplicar estrictamente el patrón 'Eager Loading' mediante dos pasos: primero, recuperar la colección principal de entidades; segundo, extraer los IDs únicos relacionados y ejecutar una única consulta secundaria utilizando la cláusula 'WHERE IN (...)' con dichos IDs agrupados. 3) Forzar el mapeo en memoria de PHP utilizando arreglos asociativos indexados por el ID de la entidad relacionada para garantizar que la hidratación final de los objetos de dominio tenga una complejidad algorítmica constante O(1), evitando loops anidados en la capa de software."
* Por qué importa: Multiplica de forma drástica la velocidad de respuesta de la PWA y reduce el consumo de recursos en Hostinger Premium. Al pasar de realizar 100 micro-consultas consecutivas a ejecutar solo 2 consultas masivas estructuradas y mapeadas por ID, el uso de la base de datos pasa a ser óptimo, permitiendo que la agenda diaria de la Estética Carolina Mora cargue en milisegundos, incluso en días con máxima saturación de turnos.
